## 🚀 Продакшен: Railway + Postgres (с 2026-07-16)

**Сайт задеплоен и живёт на общем бэкенде, не на localStorage.**
Production URL: `avesto-cup-2026-production.up.railway.app`. GitHub:
`github.com/abdusalomrabikhov-ai/avesto-cup-2026`.

Архитектура: `server/index.js` (Express) — один процесс, отдаёт и
`/api/data` (GET/PUT всего документа `TournamentData` целиком, JSONB в
Postgres), и собранную статику `dist/`. `POST /api/login` проверяет
`ADMIN_PASSWORD` (переменная окружения Railway, не хардкод) — сервер, не
клиент. Фронт (`src/data/store.ts`, `src/hooks/useTournamentData.tsx`)
дёргает эти эндпоинты вместо localStorage; публичные страницы дополнительно
поллят `GET /api/data` раз в 8 сек, чтобы видеть чужие правки без ручного
рефреша. Деплой — `git push` в `main`, Railway передеплоивает сам.

**Локальный dev-сервер (`npm run dev`, порт 3002) больше не читает
localStorage тоже** — он делает те же fetch-запросы к `/api/data`, значит
для локальной разработки нужен либо запущенный `server/index.js` с доступом
к БД, либо `VITE_API_URL`, указывающий на прод (аккуратно — PUT туда правда
уйдёт в общую базу).

**Билдер Railway — Dockerfile, не Nixpacks/Railpack.** Дефолтный
auto-detect билдер Railway (что Nixpacks, что подключённый позже Railpack)
падал на сборке с ошибкой `install apt packages: libatomic1 / secret
Reference not found` — воспроизводилось стабильно, непонятная причина на
стороне платформы, не наш конфиг (пробовали `nixpacks.toml`, `engines` в
package.json — не помогло, Railway тем более молча откатывал Builder
обратно на "Railpack (Default)" при каждом новом деплое, если не выбрать
Dockerfile явно в Settings → Build → Builder). Если когда-то потребуется
поменять раннер — учитывать, что дефолт нестабилен для этого проекта,
Dockerfile — рабочий путь.

**`ADMIN_PASSWORD` живёт только в Railway Variables**, не в git. Старый
пароль (`avesto2026`), что раньше лежал в `src/config.ts` открытым текстом
в бандле — считать скомпрометированным, `config.ts` удалён из кода
полностью.

**Первое наполнение БД сидом** — кнопка «Сбросить к исходным данным»
добавлена в `/admin/settings` специально для этого (раньше `resetTournament`
существовал в коде, но нигде не был подключён к UI).

## ⚠️ Единственный источник правды по командам (актуально до деплоя на Railway)

**Проблема (2026-07-16, до появления бэкенда):** пользователь переименовал
все 18 команд на кириллицу и загрузил логотипы через админку в своём
обычном браузере. Эти правки жили **только в localStorage** этой конкретной
сессии браузера. Открыв сайт в режиме инкогнито или на другом устройстве, он
видел старые английские названия (Zet-Mobile, Akia Avesto, Arter Avesto...)
из `src/data/seed.ts` — потому что чистый localStorage не содержал его
правок, только стартовый seed. **Это исправлено переносом названий/логотипов
в seed.ts (см. лог ниже) и полностью снято архитектурно после перехода на
общий бэкенд** — больше не актуально как риск, но раздел оставлен для
истории решения.

**Правило:** `src/data/seed.ts` — единственный источник правды на старте.
Всё, что вводится через UI админки (название, компания, лого, состав,
матчи), — временное состояние конкретного браузера, пока явно не перенесено
в код. Если пользователь хочет, чтобы правки команд «сохранились навсегда»
(видны всем, переживают инкогнито/деплой/сброс) — их нужно перенести в
`seed.ts` вручную.

**Как получить точный список команд из браузера пользователя** (у ассистента
нет доступа к его localStorage напрямую — только к своей собственной
browser-tooling сессии, которая НЕ то же самое, что браузер пользователя):

Попросить пользователя открыть свою рабочую вкладку (не инкогнито),
Cmd+Option+J (Mac Chrome) → напечатать `allow pasting` → Enter → вставить:

```js
copy(JSON.stringify(JSON.parse(localStorage.getItem('avesto-2026-tournament-data')).teams.map(t => ({id: t.id, name: t.name, company: t.company, group: t.group, hasLogo: !!t.logo})), null, 2))
```

Результат копируется в буфер — пользователь вставляет его в чат. Логотипы
(base64) через этот метод не получить — если логотип реально сменился (не
тот, что уже в `src/assets/logos/`), нужен отдельный файл от пользователя.

**Не полагаться на скриншоты/память из истории разговора** для восстановления
точных названий — риск ошибиться в написании (например «Артел Авесто» vs
«Артель Авесто»). Всегда просить свежий экспорт, если нужна 100% точность.

## UI-договорённости

- Диалоги подтверждения — только кастомные `useConfirm()`/`useToast()`
  (`src/hooks/useConfirm.tsx`, `src/hooks/useToast.tsx`), никогда нативный
  `confirm()`/`alert()`. Применено везде в админке (2026-07-16).
- Формат дат везде на сайте — ДД.ММ.ГГГГ (`src/lib/formatDate.ts`), единый
  хелпер вместо разрозненных мест с сырым ISO `YYYY-MM-DD`.
- Логотип команды — только через `<TeamLogo>` компонент, никогда не
  инлайновая `<img>`-разметка в отдельных страницах (иначе визуальная
  рассинхронизация padding между компонентами, была реальная жалоба
  пользователя на «логотипы смещены»).
- Метка «Победа» в `MatchRow` — рендерится всегда (через `invisible` класс
  у непобедителя), не условно — иначе высота карточек в списке матчей
  скачет между строками с победителем и без.

## Мобильная адаптация

- Брейкпоинт desktop-навигации в шапке — `xl:` (1280px), не `md:`/`lg:` —
  на разрешениях 1024–1279px полная навигация (6 пунктов) физически не
  помещается, обрезает кнопку «Админ».
- Sidebar-формы в админке (список + форма, `grid-cols-[1fr_320px]`) —
  переключение на `xl:`, та же причина (тесно на 1024–1279px).
- Все поля форм — минимум 16px шрифта (`input, select, textarea` в
  `index.css`) — иначе iOS Safari зумит страницу при фокусе на поле,
  независимо от `text-sm`/`text-xs` в разметке.
- Touch-таргеты кнопок-иконок в админке — минимум 44×44px
  (`min-h-11 min-w-11`), стандарт WCAG/Apple HIG.
- `viewport-fit=cover` в `index.html` + `env(safe-area-inset-*)` в
  `index.css` (`.site-header`, `.safe-bottom`, `body` left/right) — под
  «остров»/notch iPhone и системные жесты Android.
- Полный мобильный аудит проведён 2026-07-16 (делегирован фоновому агенту) —
  нашёл и починил: горизонтальный overflow от countdown-таймера на узких
  экранах, таблицы без scroll-контейнера (`GroupTable`, рейтинг
  бомбардиров), toast без учёта safe-area снизу.

## Данные и персистентность

- **Нет бэкенда.** Все данные — `localStorage`, ключ
  `avesto-2026-tournament-data` (см. `src/data/store.ts`). Первый заход
  сидит из `src/data/seed.ts`, дальше читает/пишет напрямую localStorage.
- **Производные данные не хранятся.** Турнирная таблица (`standings.ts`),
  сетка плей-офф (`playoff.ts`), рейтинг бомбардиров (`scorers.ts`) —
  всегда пересчитываются через `useMemo` из текущих matches/teams/players
  при каждом рендере. Никогда не кэшируются как отдельное поле в данных —
  так исключается рассинхронизация.
- **Изображения (логотипы/фото)** — base64, сжимаются на клиенте через
  canvas resize + WebP re-encode перед сохранением (`fileToBase64` в
  `store.ts`, `MAX_IMAGE_DIMENSION=400`, `IMAGE_QUALITY=0.85`) — квота
  localStorage общая на весь турнир сразу, без сжатия быстро упирается в
  лимит браузера.
- **Ошибка переполнения localStorage** обрабатывается явно —
  `saveData()` бросает описательную ошибку, `useTournamentData` откатывает
  состояние и показывает alert, вместо падения всего приложения в
  необработанном useEffect.
